A Practical DevSecOps Blueprint for Real Teams

Posted by Sritej Panchumarthi · DevSecOps · AWS · GitLab CI
GitLab CI Snyk Trivy SBOM Policy-as-Code

Most “DevSecOps blueprints” look great in slides and fall apart the moment a real team tries to implement them. In this article, I’m focusing on something you can actually deploy: a GitLab CI pipeline that bakes in security at every stage without turning every merge request into a two-week negotiation.

Pipeline Overview

The high-level design is simple: keep the happy path fast, and run heavier security stages in parallel or asynchronously. Everything is driven by configuration – no one should be editing YAML in production to toggle security. 

stages:
  - lint
  - test
  - build
  - scan
  - deploy

include:
  - local: .gitlab/security.yml
  - local: .gitlab/docker.yml
  - local: .gitlab/deploy.yml

Security Stages That Don’t Block Progress

The key to adoption is sensible defaults. Critical issues block the pipeline. Everything else is turned into a comment on the merge request with clear context. Developers get signal, not noise; security gets a trackable history of findings. 

sast:
  stage: scan
  allow_failure: false
  script:
    - snyk code test --severity-threshold=high

sca:
  stage: scan
  allow_failure: true
  script:
    - snyk test --severity-threshold=medium

SBOMs and Compliance

Every build publishes an SBOM artifact and pushes a signed image to the registry. This makes it trivial to answer “Where is this vulnerable library running?” when the next CVE drops.
