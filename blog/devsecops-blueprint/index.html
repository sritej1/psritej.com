A Practical DevSecOps Blueprint for Real Teams

Posted by Sritej Panchumarthi · DevSecOps · AWS · GitLab CI
GitLab CI Snyk Trivy SBOM Policy-as-Code

Most “DevSecOps blueprints” look great in slides and fall apart the moment a real team tries to implement them. In this article, I’m focusing on something you can actually deploy: a GitLab CI pipeline that bakes in security at every stage without turning every merge request into a two-week negotiation.

Pipeline Overview

The high-level design is simple: keep the happy path fast, and run heavier security stages in parallel or asynchronously. Everything is driven by configuration – no one should be editing YAML in production to toggle security. 

stages:
  - lint
  - test
  - build
  - scan
  - deploy

include:
  - local: .gitlab/security.yml
  - local: .gitlab/docker.yml
  - local: .gitlab/deploy.yml

Security Stages That Don’t Block Progress

The key to adoption is sensible defaults. Critical issues block the pipeline. Everything else is turned into a comment on the merge request with clear context. Developers get signal, not noise; security gets a trackable history of findings. 

sast:
  stage: scan
  allow_failure: false
  script:
    - snyk code test --severity-threshold=high

sca:
  stage: scan
  allow_failure: true
  script:
    - snyk test --severity-threshold=medium

SBOMs and Compliance

Every build publishes an SBOM artifact and pushes a signed image to the registry. This makes it trivial to answer “Where is this vulnerable library running?” when the next CVE drops.

Level 5: Ready-to-Build Pipeline and Policy

Drop these files into the repo to enforce security without slowing developers.

# .gitlab-ci.yml
stages: [lint, test, scan, build, deploy]
scan:
  stage: scan
  image: node:20
  script:
    - npm ci
    - npx semgrep --config p/owasp-top-ten
    - npx snyk test --severity-threshold=high
build:
  stage: build
  image: docker:26
  services: [docker:26-dind]
  script:
    - docker build -t $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA .
    - syft $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA -o spdx-json > sbom.json
    - cosign sign --yes $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA
deploy:
  stage: deploy
  script:
    - kubectl apply -f k8s/
    - kubectl rollout status deploy/app
    
# policy.yaml
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sPSPTrustedImages
metadata:
  name: require-signed-images
spec:
  parameters:
    allowedRegistries: ["registry.gitlab.com/your-group"]
    requireSignedImages: true
    verifyAttestation:
      attestors:
        - name: cosign
          keys: ["k8s://cosign-public-key"]

Ready-to-Build Steps

  • Create a .gitlab folder with shared templates for scan/build/deploy to enforce consistency.
  • Enable branch protection and require the scan job above to pass before merge.
  • Install Gatekeeper on the cluster and apply the signed-image constraint.
  • Publish SBOMs to artifact storage and wire an alert when a new CVE maps to a running digest.
